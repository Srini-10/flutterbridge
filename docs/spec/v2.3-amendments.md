# Specification v2.3 — Amendments to v2.2

**Status:** Adopted at the M2-T6 gate (2026-07-12), before N5 mints its first node.
**Scope:** Two amendments to canonical form (§2.5) and node identity (§2.3). No schema change.

---

## A15 — Canonical numeric serialization (amends §2.5)

### The conflict

§2.5 defined canonical JSON as *sorted keys, nulls omitted* — and said nothing about numbers. Both
implementations therefore did what their host language does, and their hosts disagree:

| value | Dart | JavaScript |
| --- | --- | --- |
| the double `100.0` | `100.0` | `100` |
| the double `1e20` | `100000000000000000000.0` | `100000000000000000000` |
| the double `-0.0` | `-0.0` | `0` — **the sign is silently lost** |

M2-T6 found this by round-tripping the analyzer's real output through the TypeScript loader: **26 of
1,166 lines across the C1 corpus differed, and every one was this.** Nothing else diverged — not key
order, not string escaping, not the shortest-round-trip digits, not even the exponent thresholds.

### Why it is fatal rather than cosmetic

**A node's identity is a hash of its canonical form.** Today only Dart mints ids, so the divergence is
invisible. The moment **N5 (`lift-closures`) synthesizes a `sig.Action`** — the first node ever minted
in TypeScript — the two domains would hash the same node to two different ids.

That is not a bug that produces an error. It produces a program in which the same node has two
identities: overrides miss, the cache serves the wrong artifact, incremental output stops matching a
clean build. It is the ISSUE-7 failure mode arriving through a door nobody was watching.

### The amendment

Canonical numeric serialization is defined **by the specification, not by either host language**. Both
implementations implement this; neither delegates to its host's number formatter.

A numeric value is serialized as follows.

1. **NaN and ±Infinity are prohibited.** They have no JSON representation, and a node carrying one is
   invalid. (JavaScript silently serializes `NaN` as `null`; Dart throws. Neither is acceptable.)

2. **Integers outside the exactly-representable range are prohibited** — that is, any integer with
   magnitude greater than 2⁵³−1 (`9007199254740991`). A 64-bit Dart `int` above that cannot survive a
   round-trip through an IEEE-754 double, so it cannot survive a round-trip through JSON read by
   JavaScript. It is refused at the emit boundary (`BRG1209`), not silently rounded.

3. Otherwise the value is written as the **shortest decimal string that parses back to exactly the same
   value**, subject to:

   * **No trailing `.0`.** An integral value is written as an integer: `100`, not `100.0`;
     `100000000000000000000`, not `100000000000000000000.0`. The node's `type` already records whether
     it is an `int` or a `double` — the JSON number carries the *value*, and the type carries the
     *type*. Encoding the same fact twice is how two encodings of it come to disagree.
   * **Zero is unsigned.** `-0.0` is written `0`. IEEE-754 distinguishes the two; the UIR does not, and
     JSON cannot carry the distinction anyway.
   * **Exponent form only when the shortest representation requires it**, i.e. for magnitudes ≥ 10²¹ or
     < 10⁻⁶, written lowercase with an explicit sign: `1e+21`, `1e-7`.

Rules 3's digit generation and exponent thresholds are exactly what Dart's `double.toString()` and
JavaScript's `Number.prototype.toString()` already produce — verified empirically across the range,
including subnormals (`5e-324`) and the maximum double (`1.7976931348623157e+308`). The amendment adds
only the two normalizations above, which is why it is small enough to be provable rather than merely
believed.

## A16 — Node identity is language-independent (amends §2.3)

> **A node's identity is defined by its canonical serialization, as specified in §2.5 — not by the
> implementation language that produced it.**
>
> Any two conforming implementations, given the same node, MUST produce the same canonical bytes and
> therefore the same `NodeId`.

This is what A15 exists to make true, and it is stated separately so that a future implementation — a
third language domain, a plugin, a rewriter — cannot satisfy the schema while quietly disagreeing about
identity.

## Effect on completed milestones

`x-uir-version` **1.1.0 → 1.2.0**, and `uirSchemaHash` changes with it.

That is deliberate. The canonical form is part of the UIR contract, and a change to it must invalidate
**every cached artifact and every previously computed `NodeId` exactly once**. The version context
(M1-T5) already carries the schema hash for precisely this purpose, so the invalidation is automatic
and needs no migration.

**One invalidation is enormously cheaper than permanent cross-language nondeterminism.**

| Milestone | Effect |
| --- | --- |
| M1-T2 (codegen) | `canonicalEncode` is now generated into both language models. It is the only encoder either domain may use for UIR. |
| M1-T3 (builder) | Ids are hashed from `canonicalEncode` output. **Every NodeId containing a double changes, once.** |
| M1-T4 (emitter) | Serializes through `canonicalEncode`; refuses a value A15 prohibits (`BRG1209`). |
| M1-T5 (cache) | Invalidated once, by the schema hash. Designed behaviour. |
| M2-T6 (compiler) | Loads and re-emits **byte-identical** output to the analyzer's, which is the property that was missing. |
