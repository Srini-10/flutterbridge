# Specification v2.1 — Amendments to the frozen v2.0

**Status:** Adopted at the M0-T7 gate (2026-07-12).
**Scope:** These are the *only* changes to Spec v2.0. Every one traces to evidence produced in M0-T3…T6.
Unrelated sections are untouched.

**Nothing here changes the UIR schema.** Every amendment is expressible in the frozen vocabulary. That
is the single most important sentence in this document: five spikes, four defect classes, and the IR
held.

---

## A1 — Normalization pipeline (amends §3.3)

One pass is added. **N11 `promote-cross-route-state`** (ADR-11). Full definition in ADR-11; summary:

| | |
| --- | --- |
| Position | last in normalization, after N1–N10 |
| Requires | `nav-graph` analysis |
| Invalidates | `nav-graph`, `reactivity-graph` |
| Inputs | `app.Route`, nav-graph transitions, `sig.Signal(scope: component)`, `sig.Action`, `ui.Component` params |
| Outputs | promoted signals/actions in a synthesized `app.Store{origin: "promoted"}`; route transitions stripped of state/callback arguments |
| Failure | `BRG2303` (error) → override required. Never silent. |

Updated pipeline:


## A2 — N10 emits derived Material roles (amends §3.3 N10, §5 theme engine)

Per **ADR-13**. N10 no longer merely lifts the colour literals present in the source; it emits the
**complete derived Material 3 role set** as `app.Token` nodes, computed Dart-side with
`material_color_utilities` (the package Flutter itself uses).

**Ownership split — normative:**

| Concern | Owner | Why |
| --- | --- | --- |
| Colour **palette** (role derivation from seed/primary + brightness) | **compile time** — analyzer + N10 | It is data; three realms consume it; deriving in each runtime kit would duplicate Material's algorithm N times. |
| Colour **composition** (elevation overlays, state layers, opacity blends) | **runtime kit** | These are functions of component state, which the compiler cannot know. Composed from tokens only. |

New dependency: `bridge_analyzer` → `material_color_utilities`.

## A3 — `layout-boundedness` analysis extended (amends §3.3 analyses; **no new pass**)

Evidence: M0-T4 F-layout-1. Flutter's `Center` passes *loose* constraints — the child may take up to
the parent's width and **decides for itself**. CSS has no equivalent: a flex child shrink-wraps, full
stop. One CSS rule cannot serve both `Center(child: Column(stretch))` and `Center(child: Text)`.

The existing `layout-boundedness` analysis is the right home — but as specified it only chooses a
tier. It is extended to emit, per UI node:

```
{ widthIntent: "fill" | "shrink" | "fixed",
  heightIntent: "fill" | "shrink" | "fixed",
  tier: "css" | "measured" }
```

`widthIntent` is determined by the child's own alignment (a `Column` with
`crossAxisAlignment: stretch` fills; the same Column with `center` shrink-wraps). Ui-realm generators
**must** consume it; the kit exposes the intent as a prop rather than guessing.

**No new normalization pass is required** — this is analysis output, not an IR rewrite. The **measured
tier remains post-MVP**; at MVP the CSS tier plus an explicit intent covers the observed cases.
Calibration at **M2-T12**.

## A5 — Verification (amends §8)

**The authoritative visual metric is VMR** (ADR-12): pixelmatch, YIQ threshold `0.1`, `includeAA:
false`, `mismatched / total`. Provisional gate **3 %** per route × viewport; **re-calibrated at
M2-T12**.

**New verification rule VR-1 — token parity.** Colour correctness is checked as **data, not pixels**:
the compiler-derived token role set vs the roles Flutter actually resolves. Failure = `BRG4101`, hard,
independent of VMR. Mechanism (reference probe vs `material_color_utilities` comparison) decided at
**M4-T1**.

**Reporting is plural by mandate.** Every visual report prints VMR (gating) *and* exact mismatch
(diagnostic) *and* the flat-region colour table. **No report may present a single number** — the
one-number report is the defect M0-T5 found.

## A7 — Compatibility report policy (formalizes the M0-T6 prototype)

**Descriptive bands. No numeric score. The reasoning is the product; counts are only evidence for it.**

| Band | Criteria |
| --- | --- |
| **High** | Zero unsupported constructs. Everything is MVP or Partial. Warnings may exist, but each has a documented resolution. |
| **Medium** | Unsupported constructs exist, but **all are overridable leaves** (CustomPainter, explicit animation, transitions). Bounded per-widget cost. No platform-capability gaps. |
| **Low** | At least one **platform-capability gap** — platform channels, isolates, `dart:io`, `dart:ffi`, native views — **or** unsupported constructs on the application's critical path. |

The report **must** distinguish *"a gap in our compiler"* from *"a gap in the browser"*. A blanket
"Low" that does not say which is useless to a developer. (Evidence: the M0-T6 `shop_bridge` verdict
text is the model.)

## A8 — Construct status taxonomy (new; governs the catalog for all future versions)

Every construct the compiler knows about carries **exactly one** status, held as data in the
catalog/adapter registry:

| Status | Meaning | Guarantees | Diagnostic |
| --- | --- | --- | --- |
| **Supported** | Mapped and covered by G1/G2 goldens + conformance. | Stability guarantees apply. | — |
| **Partial** | Recognised; mapping exists but is incomplete or behind a documented gap. | No completeness guarantee. | warning |
| **Experimental** | Mapping exists behind an opt-in config flag. | **May change without a major.** | warning |
| **Deprecated** | Was Supported; removal scheduled. | Removed at next major, **with a codemod**. | warning |
| **Unsupported** | Recognised; no mapping and none planned (platform gap). | → `Opaque` + override required. | error |
| **Unknown** | In **no** catalog. | → `Opaque` + telemetry. | `BRG2102` |

**Rules.**
1. **Unknown is a statement about our catalog, not about the construct.** It is the primary input to
   adapter-registry prioritisation (and to the AI gap analyzer at M7).
2. A user-declared widget is **never** Unknown — it is the thing being compiled. *(Evidence: the
   M0-T6 prototype's own false positive, which reported the user's screens as unknown constructs.)*
3. Promotion Unknown/Partial → Supported requires goldens.
4. Supported → Deprecated requires a codemod before the removing major.
