# Spec v2.5 — Amendments

Amends Spec v2.0 (frozen) as extended by v2.1, v2.2, v2.3 and v2.4.

Status: **approved**. Ruling given at the M3-B gate, on the evidence in §A18.1.

---

## §A18 — An action may take parameters

### A18.1 The evidence

`FavoritesStore.toggle(int id)` is the second thing `fixtures/apps/hello_bridge` does, and it is what
`sig.Action`'s own documentation already cites as its motivating example:

> C1 evidence: `FavoritesStore.toggle` mutates via `_favoriteIds.add/remove`, so an assignment-only analysis
> returns an empty write set — and generated React state that never updates.

The write set was fixed. The parameter was not noticed, because until M3-B nothing consumed an action body.
The first generator to try it found this:

```json
{"kind":"sig.Action","id":"…","writes":["…"],"body":[{"kind":"logic.If","test":{…}}]}
```

The body reads `id`. **Nothing declares it.** `sig.Action`'s complete field set is `anchor`, `body`, `ext`,
`id`, `isAsync`, `kind`, `span`, `writes`.

Measured on the corpus:

| shape | where | parameters |
|---|---|---|
| `void toggle(int id)` | `hello_bridge`, `favorites_store.dart` | one positional |
| `onChanged: (value) => setState(…)` | `hello_bridge`, `login_screen.dart` ×2 | one positional, via an N5-lifted closure |
| `void _toggleTheme()` | `hello_bridge`, `main.dart` | none |

So both shapes are real: an action with parameters, and an action without. The second is why `params` must be
optional rather than an empty list — see §A18.3.

### A18.2 The contradiction

A `logic.Ref` to a parameter is, in the frozen model, **indistinguishable from a reference to anything else
that is not in the program**: a top-level function, a package API, a typo. `Ref.target` is *"the declaration
referred to, when it is in the program"* — and a parameter is not in the program, because there is nowhere to
put it.

The consequence is not cosmetic. A generator has exactly two options, and both are wrong:

1. **Emit the bare name.** `_favoriteIds.contains(id)` compiles to a file where `id` is undefined. The
   application fails at the first click.
2. **Infer the parameters** from the unresolved names the body reads, and synthesise a signature. This fails
   on the first store that calls a top-level function — it would emit `action((notifyListeners) => …)` — and
   it fails *silently*. It also cannot recover order, type, optionality, or named-vs-positional, every one of
   which the call site needs.

So **no parameterised action can be generated, for any target**, and every real store has one. The React
generator reports `BRG3006` and emits nothing, which is correct and is also the end of the road.

The precedent is exact, and this is its third instance. §A10 amended the model when it genuinely lacked a
representation for `logic.Assign`. §A17 did it again for `app.RouteTransition`, on the finding that *"the
reason it was never emitted is not an oversight. It is that the frozen schema cannot describe what real
Flutter code does."* Each was found the same way: by pointing the tool at a real program and reading what came
out.

### A18.3 The amendment

`sig.Action` gains one optional field.

| field | | means |
|---|---|---|
| `params` | `ParamDecl[]`, new, optional | The action's parameters, in order. |

**`ParamDecl` is reused, not redefined.** It already exists in `l1.json` and already carries `name`, `type`,
`required`, `named` and `defaultValue` — it is what `ui.Component.params` and `app.Route.params` are made of.
A second parameter model would be a fact stated twice, which is what ADR-18 exists to prevent, and there is
nothing about an action's parameter that a component's parameter does not already have.

**Absent means the action takes none.** Not `[]`. The distinction is load-bearing for two reasons:

- Every existing `sig.Action` stays byte-identical, so its content-addressed id is unchanged. Emitting `[]`
  for the zero-parameter case would re-hash every action in every cached document, for no new information.
- It matches the model's existing convention — `Store.signals`, `Component.params` and `Route.params` are all
  optional in exactly this way.

**Ids are unaffected — including for actions that gain parameters.** This was expected to be a cost and is
not, and the reason is ADR-17's split, working exactly as designed:

- An action the **analyzer** emits is a *declaration*, and `builder/node_factory.dart` derives a
  declaration's id from its **symbol**, never its content: *"Editing the body of a method must not change the
  id of the signal it writes."* Adding `params` changes the content and not the symbol, so the id stands.
  Measured on `hello_bridge`: two of thirty nodes changed, both only by gaining `params`, and **all thirty ids
  are byte-identical**.
- An action **N5 synthesizes** from a closure has no symbol — there was no method — so it is content-addressed
  via `nodeIdOfContent`, and one that carries parameters does get a new id. That is correct and costs nothing:
  such an action's body previously referenced a name nothing declared, so no document minted before this
  amendment carried it correctly, and there is nothing to preserve.

**Resolution is by name, and this is not an inference.** A `ParamDecl` has no `id` — it is a value, not a
node, and carries no `UirNodeBase`. So a `logic.Ref` in an action body resolves to a parameter by matching
`Ref.name` against the action's declared `params`, within the action's scope. That is ordinary lexical
scoping, and it is exactly how `ui.Component.params` already works (`bind.Param` names its parameter; it does
not point at it). A name that matches no parameter and no declaration is still unresolved, and still an error
— the amendment removes the false negatives, not the check.

### A18.4 What this does not change

- **`sig.Derived` gains nothing.** It is *"a value computed from other signals — a Flutter getter over
  state"*, and a getter takes no arguments. This one survives contact with the corpus.
- **`sig.Effect` gains nothing — and this claim was wrong.** It was written here as *"its `timing` says it is
  invoked by the component lifecycle, not by a caller with arguments in hand"*, which sounds right and is
  false: `didUpdateWidget(Screen oldWidget)` is invoked by the lifecycle **with an argument**, and
  `catalog/widgets/material.json` maps `didUpdateWidget → update`, so it becomes a `sig.Effect` whose body
  reads `oldWidget` and whose schema has nowhere to declare it. That is verbatim §A18's defect, one node kind
  over. It was found the way §A18 was — by running the analyzer, not by reading the schema — and it is
  **deliberately not amended here**: see [the §A19 gap](../m3/m3b1-sig-effect-params-gap.md). The sentence is
  left in the record rather than quietly corrected, because *"a getter takes no arguments"* and *"a lifecycle
  callback takes no arguments"* are the same shape of reasoning, and only one of them is true.
- **No pass changes semantics.** N1–N11 rewrite bodies and lift closures; none of them removes a declaration,
  and `params` is carried like any other field. N5 is the one to watch, because it *creates* actions from
  closures — a lifted `(value) => setState(…)` must arrive with `value` declared, or this amendment fixes the
  store case and leaves the widget case exactly as broken.
- **The runtime needs no change.** `@bridge/runtime-react`'s facade is already
  `action<A extends readonly unknown[], R>(body: (...args: A) => R)`; it forwards arguments, batches the call,
  and reports them to `onAction`. That is not luck — an action is a function, and it was typed as one.

### A18.5 Version

`x-uir-version`: **1.3.0 → 1.4.0**. Minor: the change is purely additive and invalidates no existing node
shape.

`UIR_SCHEMA_HASH` changes to `fc4e4eb130c9f948`, as it must — the hash is taken over the raw schema bytes, so
every consumer is forced to re-read the model rather than half-trust a stale one (INV-5).

### A18.6 Why the parameter was invisible for four milestones

Worth recording, because it is the same lesson twice in one week.

The schema validated. The types compiled. The analyzer's tests passed, the compiler's 115 tests passed, and
the model round-tripped through canonical JSON byte-for-byte. Every gate the project has was green, and the
model could not express a Flutter method with an argument.

It was invisible because **nothing had ever read an action body**. The analyzer wrote it, the compiler carried
it, and the first component to look inside one was a generator that did not exist until M3-B. A field is only
tested by the thing that consumes it, and until something does, "the schema has it" and "the schema is right"
are indistinguishable.

This is the second defect in one milestone found by pointing the tool at one real 369-line application — the
first being ADR-21's colour encoding, which every hand-written fixture had agreed with because the same person
wrote both. `hello_bridge` is 369 lines. `compass_app` and `wonderous`, which the corpus reports measure but
this repository does not contain, are 9k and 30k.
