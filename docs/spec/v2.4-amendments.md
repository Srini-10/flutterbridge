# Spec v2.4 — Amendments

Amends Spec v2.0 (frozen) as extended by v2.1, v2.2 and v2.3.

Status: **approved**. Ruling given at the M3 gate, on the evidence in §A17.1.

---

## §A17 — A route transition may name a destination that is not a route

### A17.1 The evidence

`app.RouteTransition` has existed since v2.0 and **the analyzer has never emitted one**. The nav graph
therefore has routes and no edges, N11 (`promote-cross-route-state`) has never fired on any real program,
and no generator can emit navigation. It is the one item gating M3.

The reason it was never emitted is not an oversight. It is that the frozen schema cannot describe what
real Flutter code does. Measured across the corpus:

| shape | where | destination |
|---|---|---|
| `Navigator.push(context, MaterialPageRoute(builder: (_) => HomeScreen(…)))` | `hello_bridge` ×1 | an **inline component** — no path, no name |
| `context.go('/path')` | `wonderous` ×6 (currently swallowed by `ui.Opaque`) | a **declared route** |
| `Navigator.pop()` | `wonderous` ×1 | no destination at all |
| `GoRoute(path: …)` declarations | `compass_app` ×36, `wonderous` ×46 | — (declarations, already extracted) |

Both destination shapes are real, and they are not the same kind of thing.

### A17.2 The contradiction

`app.RouteTransition.target` is **required**, is a `NodeId`, and its description reads *"The destination
route."*

`app.Route` **requires** `path` — a URL string.

For `Navigator.push(context, MaterialPageRoute(builder: (_) => HomeScreen()))` the destination has **no
URL**. It is a widget constructed inline and pushed onto a stack. Therefore:

- no `app.Route` node can exist for it (there is no `path` to give), so
- `target` — required, and meaning *route* — has nothing legal to point at.

The four ways out, and why three were refused:

1. **Point `target` at the `ui.Component`.** The schema still validates — a `NodeId` is a `NodeId` — while
   the field's own description becomes false. Every consumer written against *"the destination route"*
   breaks silently. Refused: a contract that validates while being wrong is the failure mode this project
   exists to avoid.
2. **Synthesize an `app.Route` with an invented path** (`/_push/HomeScreen`). Refused: it invents a URL the
   developer never wrote, and it would appear in the generated application. Which URL an anonymous push
   *becomes* on a path-based target like Next.js is a **legalization** decision, not an extraction one.
   Deciding it here would bury a routing policy inside the analyzer.
3. **Emit nothing and diagnose.** Honest, but it makes `hello_bridge`'s only navigation permanently
   unrepresentable, and leaves N11 unable to promote across a push boundary.
4. **Amend the model.** ✅ **Approved.**

The precedent is §A10: when the frozen model genuinely lacked a representation for something real
(`logic.Assign`), the model was amended rather than the truth distorted. This is the same situation.

### A17.3 The amendment

`app.RouteTransition` gains a second destination field. **Exactly one of the two is present.**

| field | | means |
|---|---|---|
| `target` | `NodeId`, **now optional** | The destination `app.Route`, when the navigation names one — `context.go('/wonder/3')`, `Navigator.pushNamed('/home')`. |
| `component` | `NodeId`, new, optional | The `ui.Component` rendered, when the navigation constructs its destination inline — `Navigator.push(MaterialPageRoute(builder: …))`. There is no path, and none is invented. |

Everything else is unchanged: `source`, `arguments`, and the node base.

`target` becoming optional is a contract change. It costs nothing today: **zero `app.RouteTransition`
nodes have ever been emitted**, so no document, cache entry or anchor depends on the old shape.

A transition with **no destination at all** — `Navigator.pop()` — is **not** an `app.RouteTransition`. It
is a return along an edge that already exists, not a new edge, and the nav graph does not gain anything by
inventing a node for it. Popping is left to the generator, which knows its own target's history model.

### A17.4 Why the exclusivity is not in the schema

**The UIR schema dialect cannot express it, and the amendment does not pretend otherwise.**

The dialect admits exactly four forms (`tools/schema-codegen/src/parser.ts`): an object, a `oneOf` union
that **requires** `x-uir-union: true` with `$ref`-only variants, an enum, or a string alias. There is no
`dependentRequired`, no `if`/`then`, no `not`. A "exactly one of these two properties" constraint has no
expression, and the parser rejects anything outside the dialect rather than ignoring it.

There is a deeper reason it could not work anyway. **`target` and `component` are references.** A `NodeId`
is a string; nothing about its shape says what it points at. The schema has never been able to say
"`target` points at a route" — the existing claim was documentary all along, checked (if at all) at link
time by `_checkReferencesResolve`.

So the exclusivity is enforced **in code**, which is where this project already keeps every invariant the
schema cannot state:

- *ids are functional* — `builder/validation.dart#_checkIdsAreFunctional`
- *references resolve* — `emit/validation.dart#_checkReferencesResolve`
- *required spans* — `builder/validation.dart#_checkRequiredSpans`

Added to that list:

- **`BRG1307`** — an `app.RouteTransition` must carry exactly one of `target` / `component`, and the node
  it names must be an `app.Route` / a `ui.Component` respectively. Checked at emit in the analyzer, and
  again at load in the compiler: a document from another frontend is not trusted to have got it right.

### A17.5 Version

`x-uir-version`: **1.2.0 → 1.3.0**. Minor: the change is additive plus one relaxation, and no existing
node shape is invalidated.

`UIR_SCHEMA_HASH` changes, as it must — the hash is taken over the raw schema bytes, so every consumer is
forced to re-read the model rather than half-trust a stale one (INV-5).

### A17.6 What this does not decide

**How an inline destination becomes a URL on a path-based target is out of scope**, and deliberately so.
A `component` destination says only what the Flutter program does: it renders that component, with those
arguments, on a new stack entry. Whether the React/Next.js generator gives it a synthetic route, a modal,
or a client-side stack is a legalization decision, to be made in the layer that knows the target — with
the evidence in front of it, not guessed at here.
