# Specification v2.2 тАФ Amendments to v2.1

**Status:** Adopted at the M1-T8 gate (2026-07-12), before any extraction code was written.
**Scope:** Three amendments. Two change the UIR schema; one clarifies wording.

**This is the first time the UIR schema has changed.** v2.1 opened with the sentence *"Nothing here
changes the UIR schemaтАж five spikes, four defect classes, and the IR held."* It no longer holds, and
the reason is worth stating plainly rather than burying: **the IR could not represent assignment.**
Five spikes did not find it because every one of them measured *widgets*. The gap is on the axis
nobody had measured тАФ Dart statements тАФ and it was found by the first milestone that had to *produce*
statements rather than count them.

---

## A10 тАФ `logic.Assign` (amends the UIR `Expr` union)

### The conflict

The `Expr` union had 17 variants and **none of them was an assignment**. `PropertyAccess` is
explicitly *"Reading a property of a value."* There was no `=`, no `+=`, no `++`.

The consequence is not cosmetic. `sig.Action.body` is `List<Stmt>` тАФ the normalized form of a
`setState` body тАФ so `setState(() => _count++)` could only be extracted as:

```
sig.Action(writes: [_count], body: [ExprStmt(OpaqueExpr("_count++"))])
```

`writes` correctly records **that** `_count` changes. Nothing in the UIR records **what it becomes**:
the body is a Dart source string. No generator can compile that without parsing Dart, which is the one
thing the UIR exists to prevent.

The failure would have been silent. `OpaqueExpr` is structurally legal, so the schema validator would
pass, extraction would "succeed", and CI would be green тАФ and the defect would surface in M2 as
*"generated React state never updates."* That is the exact failure `sig.Action`'s own description
warns about, one axis over.

### The evidence

A construct census of the C1 corpus (parse-only, against the 57 frozen kinds):

| | hello_bridge | compass_app | wonderous |
| --- | --- | --- | --- |
| Assignments | 8 | 60 | 333 |
| тАФ inside `State` / `ChangeNotifier` / `Notifier` | **8 (100 %)** | 34 | 176 |

**Every state mutation in all three applications is an assignment.** In our own fixture, every
assignment is a state mutation. This is not a tail case; it is the thing the compiler is for.

### The amendment

```
logic.Assign            (added to the Expr union, after Unary)
  target     Expr                 the lvalue written to
  operator   AssignmentOperator   required
  value      Expr?                absent for ++ / --
  isPostfix  boolean?             i++ vs ++i
  type       TypeRef              required
```

`AssignmentOperator` is an **enum, not a free-form string** тАФ the full Dart set, 16 values. This is the
one place the schema deliberately diverges from `Binary.operator`, which *is* a string. The asymmetry
is the point: a binary operator is pure, so the worst a wrong one does is compute the wrong number; a
wrong **assignment** operator writes the wrong value to state. A generator handed an operator it does
not know must fail to compile rather than guess.

`target` is an `Expr` and not a `Binary.left`, because an lvalue is not a value. A generator reading
one as the other emits a comparison where a write was meant.

### Rejected alternatives

| | Why not |
| --- | --- |
| `Binary(operator: "=")` | `Binary.operator` is a free string, so this *compiles*. It also makes `logic.Binary` mean two different things, and any generator that does not special-case it emits `count === 1` where `count = 1` was meant тАФ wrong code, no error. |
| Leave assignments as `OpaqueExpr` | Structurally legal, so CI stays green and the failure is deferred to M2 at maximum cost. |
| Rely on `sig.Action.writes` alone | `writes` is a **data-flow summary** (*which* signals change). `Assign` is **program semantics** (*what they become*). Neither replaces the other; both are required. |

## A11 тАФ `logic.OpaqueDecl` (amends the UIR `Decl` union)

The `Decl` union тАФ `ClassDecl | EnumDecl | TypeAliasDecl | FunctionDecl | FieldDecl` тАФ had **no opaque
variant**, unlike `Expr` and `Stmt`. A mixin or an extension therefore had nowhere to go and would have
had to be **silently discarded**, violating INV-4 тАФ the very invariant `OpaqueStmt` exists to satisfy.

compass_app declares 11 mixins; wonderous declares 5 extensions.

```
logic.OpaqueDecl        (added to the Decl union)
  dartSource  string
  reason      string
```

Mirrors `OpaqueExpr` / `OpaqueStmt` exactly. Later milestones replace these incrementally.

## A12 тАФ The `RawNode` contract (clarification only; no code change)

Closes **ISSUE-8**. The record model is confirmed as built; the wording is what was wrong.

*"Every RawNode must contain span, symbol, kind, references"* describes the **record graph as a
whole**, not every individual record.

* **`symbol` is optional.** Only declaration-like records own one. An expression inside a widget tree
  is not something anything refers to, and synthesizing a symbol for `1 + 2` to satisfy a sentence
  would be inventing information (INV-4).
* **References stay embedded as `RawRef` values inside `fields`.** No top-level `references` array.
  The canonical builder resolves references *structurally*, with zero per-kind knowledge тАФ that is what
  makes it kind-agnostic, and a parallel array would break it.

## A13 тАФ Node identity is two-tier (amends ┬з2.3)

Ratified at the M1-T8 gate; full reasoning in **ADR-17**.

┬з2.3 said an id is "blake3 of the node's canonical form". Taken literally that has no fixed point: an
L3 declaration and the nodes referring to it are mutually referential. The architecture is, and has
been since M1-T3:

| | Identity |
| --- | --- |
| **Declarations** | the **symbol** тАФ which is what lets a rebuilt file resolve a reference into a *cached* file to the id that file already has |
| **Tree nodes** | **content**, stripped of `id`/`anchor`/`span`, **recursively** |
| **Occurrence** | the **anchor** тАФ two identical `SizedBox`es are one node in two places |

The digest remains **SHA-256**, not blake3 (ISSUE-7): cross-language determinism outranks speed, and
hashing has never been the bottleneck.

## A14 тАФ The `ModuleExtractor` seam is asynchronous (amends ┬з7.2)

M1-T5 declared the incremental extractor seam synchronous. Extraction needs a **resolved** unit, and
`package:analyzer` resolves only asynchronously тАФ so a synchronous seam is a seam nothing can plug
into.

```
typedef ModuleExtractor = Future<List<RawNode>> Function(String path, String source);
```

**A correction to the signature, not to the algorithm.** Digests, the API/impl fingerprint split, the
cache keys and the merge order are untouched тАФ which is why `incremental тЙб clean` survived the change
rather than having to be re-proved from scratch. It is now proved end to end against a real Flutter
application (ADR-17).
