# FlutterBridge v0.1.0 — release report

**Recommendation: release.** Tag `v0.1.0` on `release/v0.1.0`, publish eight npm packages, and ship the
platform table exactly as written.

---

## 1. Executive summary

FlutterBridge compiles a Flutter application into a React/Next.js project. `npm install -g @bridge/cli`,
then `bridge build` in a Flutter project directory.

The design commitment that shapes everything else: **it refuses what it cannot translate faithfully.**
Generation stops rather than emitting a partial project, and every refusal names a capability and the
subsystem that owns it. A generated application that renders and is subtly wrong is worth less than one
that says exactly what it could not do.

What this release is:

- **A working compiler for the surface it claims.** 90 widgets render; `examples/counter` compiles with
  zero diagnostics, builds with `next build`, hydrates, and updates on interaction in a real browser.
- **Installable by someone who has never seen this repository.** Validated by a real registry
  publish-and-install cycle, not by a workspace link.
- **Reproducible.** Byte-identical output across three complete pipeline runs, and the CI matrix checks
  byte-identity *across operating systems*.

What it is not: a compiler that can emit a large production application. That remains the top known
limitation and is why the version is `0.x`.

## 2. Architecture summary

```text
Flutter project → analyzer (Dart) → UIR → N1–N11 → generator plugin → React/Next.js project
                                                         ↓
                                                   runtime kit
```

| Layer | Package | Decision |
| --- | --- | --- |
| Frontend | `bridge_analyzer` (Dart) | ADR-2 — extraction runs in Dart, against the resolved element model. Recognition is by **supertype, never by name** (C1 found 18 widgets misclassified by name). |
| IR | `@bridge/uir` | ADR-1 — layered realms (`ui.*`, `logic.*`, `bind.*`, `sig.*`, `app.*`). NDJSON on the wire (ADR-7), content-addressed nodes (ADR-17). |
| Reactivity | `sig.*` + runtime graph | ADR-4/ADR-20 — one signal model. *A signal write **is** the notification.* |
| Compiler | `@bridge/compiler` | Eleven target-neutral passes. Deterministic, idempotent: `normalize(normalize(x)) == normalize(x)`, checked on the user's own project by `bridge validate`. |
| Targets | `@bridge/gen-react` | ADR-22 — a pure, synchronous `generate` returning `EmittedFile[]`. No filesystem, clock or randomness, which is what makes determinism testable by calling it twice. |
| Runtime | `@bridge/runtime-react` | ADR-6/ADR-19 — a kit that consumes descriptors, never UIR. |
| CLI | `@bridge/cli` | INV-17 — contains no compilation logic; every stage delegates. |

Two invariants did most of the work in review: **INV-20** (the compiler owns the palette — no literal
colour survives anywhere, so a widget painting an undefined role is refused at build time rather than
throwing in a browser) and **ADR-18** (framework metadata is authored once in `catalog/widgets/*.json` and
generated into both language domains).

### Version compatibility

- Compiler train at `0.1.0`, pinned to each other **exactly** — npm cannot construct a mixed install.
- `@bridge/runtime-react` versions independently (Spec §1.4 LTS policy); emitted apps get `^0.1.0`,
  declared once and shared with `TargetGenerator.runtimeRange`.
- UIR schema `1.4.0`, hash `fc4e4eb130c9f948`, enforced: a document from a different schema is **refused**
  with exit 3, not read hopefully.
- Node ≥ 20 · Dart `>=3.11 <4.0` · React `^19` peer · Next 15.5.x (ADR-16).

## 7. Defect found during this release preparation

Running the documented commands **one at a time** rather than through `build` surfaced a release blocker:

> `bridge analyze && bridge generate` failed. `analyze` writes `uir.ndjson`; `generate` fell back to it;
> un-normalized UIR has no `app.Token` nodes because N10 has not run. Every themed widget was refused with
> BRG3010 — whose message says *"Give the app a `ColorScheme.fromSeed(...)`"*, advice the example
> application had already taken.

The compiler was telling a user to fix something that was not broken. Fixed: `generate` normalizes when
handed a raw document, through the same helper `build` uses. The regression check asserts
`analyze + generate` is byte-identical to `build`, and lives in the script that drives the real CLI —
a unit test would have had to re-implement the rule it was checking.

**This is the argument for rehearsing the documented sequence rather than the convenient one.**

## 8. Platform support

| Platform | Status | Evidence |
| --- | --- | --- |
| **macOS arm64** | ✅ **validated** | full rehearsal: registry install → build → `next build` → browser → validate |
| **Linux x64/arm64** | ⚙️ **CI-qualified, not hand-run** | the release matrix runs the full pipeline here |
| **Windows x64** | ⚙️ **fixed, CI-qualified, never hand-run** | was broken four ways through development; all fixed with tests |
| macOS Intel | ⚙️ same as Linux | same toolchain |

The ✅/⚙️ distinction is deliberate. **A green CI run is evidence; it is not the same evidence as somebody
using the thing.** An earlier edition of this table said Windows was "expected to work" when it could not
work at all — the lesson of the last two milestones is that the failure mode is not shipping with gaps, it
is shipping with gaps described as coverage.

The release matrix has been **authored and syntax-validated, and never executed on a runner** — this
environment has no Docker, no VM and no `gh` CLI. Running it once is the highest-value action after this
release.

## 9. Known limitations

1. **A large production application cannot emit.** `hello_bridge` reports 30 errors — 19 from one root
   cause (literal theme colours, so no full Material role set) and 11 genuine gaps: class emission,
   `widget`/`context` framework primitives (INV-22), `ui.Async`. This is why it is `0.x`.
2. **No incremental build.** 99% of build time is a full re-analysis. The analyzer has a `--cache` flag
   and a cache-key implementation; nothing in the CLI uses them.
3. **Imperative navigation is not implemented** — ADR-0024 is proposed, not built. Nor is the gesture
   model, explicit animation, slivers, or custom painting.
4. **Store reactivity is unvalidated in a browser.** M5-D's defect was in the component-local path; the
   store path is analogous and has never been run.
5. **One application, one browser** in the e2e suite.
6. **~1.2 GB peak analyzer memory**, a fixed cost regardless of application size.
7. **A user's CRLF project** gets different `span.length` from a colleague's LF one. Emitted application
   identical; caches not shared. One line of `.gitattributes` avoids it.

## 11. Suggested git tag

```bash
git tag -a v0.1.0 -m "FlutterBridge v0.1.0 — first public release"
```

Annotated, not lightweight: this is a release, and the tag should carry a message and a tagger.

**Note on existing tags.** `compiler-v1` and `v0.3.0-dev-preview` already exist. `v0.3.0-dev-preview`
points at `db89430`, the same commit `release/v0.1.0` branches from, and its version number is *higher*
than this release — it predates the current scheme and will confuse release tooling that sorts tags.
Recommend deleting or renaming it, as a decision for the repository owner rather than something to do
unasked.

## 13. Recommendation

**Release.**

The compiler does what it says on the surface it claims, refuses the rest legibly, installs from a
registry, produces an application that runs correctly in a browser, and reproduces byte-for-byte. The
limitations are real, bounded, and documented on the page a user reads before installing.

Two things should happen before `git push`, and neither is engineering:

1. **Decide about `v0.3.0-dev-preview`** (§11).
2. **Run the release matrix once.** It is the difference between ⚙️ and ✅ for two of four platforms, and
   everything needed for it is in this branch.
