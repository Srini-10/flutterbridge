## 2. CI

Two workflows, and both found real defects on their first real run.

### `ci.yml` — green

| Run | Result |
| --- | --- |
| PR #1, first attempt | ❌ **failed** |
| PR #1, after fix | ✅ all 4 jobs |
| push to `main` | ✅ all 4 jobs |

**Root cause of the failure**, and it was a genuine bug rather than an environment quirk:
`codegen:check` invoked `tools/widget-support-doc.mjs`, which imports the React generator's **built
output** to read `WIDGET_MAP` — but `ci.yml` ran `codegen:check` *before* `build`. Every local run passed
because `dist/` was already there from an earlier build.

```text
Error [ERR_MODULE_NOT_FOUND]: Cannot find module
  '…/packages/generators/react/dist/index.js'
  imported from '…/tools/widget-support-doc.mjs'
```

Fixed by making the script self-sufficient — it now builds `@bridge/gen-react` first, as its other three
steps already build what *they* need. Reordering the workflow would have fixed CI and left
`pnpm run codegen:check` broken on a fresh clone. Verified by deleting `dist/` and re-running.

### `release-qualification` — Linux and macOS green, Windows found a defect

First execution ever. It had been authored, syntax-validated, and every platform-independent step run
locally — and none of that could have caught what it found.

| Job | Result |
| --- | --- |
| `ubuntu-latest · pipeline` | ✅ |
| `macos-latest · pipeline` | ✅ |
| `windows-latest · pipeline` | ⏳ **hung indefinitely**, cancelled |

**Root cause:** the `toolchain versions` step ran `flutter --version | head -1`. `head` closes the pipe
after the first line; `flutter` on Windows is a batch script that does not handle the broken pipe, and the
job sat there until cancelled. Ubuntu and macOS passed the same step in seconds.

Fixed (`d44d9f1`) by removing the pipe. A re-run was dispatched.

**One Windows result did land, and it matters:** the `line endings are LF on every platform` step
**passed**. `.gitattributes` — added in M5-F specifically for Windows, and never testable before — works on
the platform it was written for.

## 4. What is not verified

Stated plainly:

- **Windows end-to-end.** The matrix re-run was dispatched but had not finished when this was written. The
  hang was in a diagnostic step, not in the compiler, and the steps before it passed — but that is a
  reason for optimism, not evidence.
- **npm publishing.** Deliberately not done: it needs an npm account and a decision about claiming the
  `@bridge` scope. The tarballs are attached to the GitHub Release; nothing installs from npm yet, so
  `docs/guide/installation.md`'s `npm install -g @bridge/cli` will not work for a stranger until it is.
- **The platform table still says ⚙️** for Linux and Windows. Linux is now genuinely CI-green and could be
  upgraded; Windows should wait for the re-run. Leaving both until the matrix is fully green keeps the
  table's meaning intact.

## 5. Assessment

**The release is real and the process worked.** Both workflows failed on first contact, both failures were
genuine defects in this project rather than infrastructure noise, and both were fixed rather than worked
around. That is the outcome a release rehearsal is supposed to produce.

The most useful thing learned: **two of the three defects found today were invisible to every form of local
validation** — one needed a clean checkout, the other needed Windows. The remaining gap in confidence is
almost entirely about environments nobody has run in, which is precisely what the matrix now covers.
