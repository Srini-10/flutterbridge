# FlutterBridge v0.1.0

**Compile a Flutter application to React.** First public release.

```bash
npm install -g @bridge/cli
cd my-flutter-app
bridge doctor && bridge build
```

You need a Flutter SDK — it bundles the Dart the analyzer runs on. Nothing else.

---

## What it does

Reads your Flutter app through Dart's **resolved element model** — types, not text — lowers it to a layered
intermediate representation, normalises it with eleven target-neutral passes, and emits a Next.js project.

**It refuses what it cannot translate faithfully.** Generation stops rather than emitting a partial
project, and every refusal names a capability and the subsystem that owns it:

```text
error [BRG3010] `Scaffold` paints the Material role `surface`, and this program's theme defines no
token for it. INV-20 requires every colour a mapped widget paints to resolve to an app.Token, so there
is no default to fall back to — a literal here would be the compiler bug that invariant names. Give the
app a `ColorScheme.fromSeed(...)`, which makes N10 derive the full role set.
```

An application that renders and is subtly wrong is worth less than one that says exactly what it could
not do.

## What works

- **90 Flutter widgets** render — layout, Material components, navigation shells, forms and selection
  controls, lazy lists, implicit animations. [Full list](docs/guide/supported-widgets.md), generated from
  the compiler's own map.
- **State and reactivity.** `setState` and `ChangeNotifier` lower to a signal graph; a signal write *is*
  the notification.
- **Theming.** `ColorScheme.fromSeed` derives all 46 Material 3 roles. No literal colour survives
  anywhere — the compiler owns the palette.
- **Routing**, asset manifests, and a runtime kit the emitted code imports.
- **Deterministic output.** The same source produces byte-identical UIR across complete rebuilds, and
  `bridge validate` checks that on *your* project.

## What does not

Stated plainly, because you will hit these:

- **A large production application cannot emit yet.** Framework primitives (`context`, `widget`,
  `mounted`), user-defined classes as values, and `FutureBuilder`/`StreamBuilder` are unmodelled.
- **Imperative navigation** (`Navigator.push`), the **gesture model**, **slivers**, **explicit
  animation** and **custom painting** are not implemented. Slivers are refused, never emulated.
- **No incremental build.** ~9 s per build, 99% of it Dart VM startup.

[Troubleshooting](docs/troubleshooting.md) is ordered by how often real applications hit each one.

## Platform support

| Platform | Status |
| --- | --- |
| **macOS (Apple silicon)** | ✅ validated end to end |
| Linux x64/arm64 | ⚙️ CI-qualified, not hand-run |
| Windows x64 | ⚙️ fixed and CI-qualified, never hand-run |

That distinction is deliberate: a green CI run is evidence, but not the same evidence as somebody using
it. If you are on Windows or Linux, `bridge doctor` is the first thing to run — and a report either way is
welcome.

## Validation

951 tests (693 TypeScript, 240 Dart, 18 browser). The browser suite generates a real application through
the real pipeline on every run — `bridge build` → `npm install` → `next build` — then drives production
*and* development servers, because React reports hydration and hook problems only in development.

`examples/counter` compiles with **zero diagnostics**, builds, hydrates, and updates when you click it.

## Packages

| | |
| --- | --- |
| `@bridge/cli` | the `bridge` command, and the Dart analyzer |
| `@bridge/compiler` · `@bridge/core` · `@bridge/uir` · `@bridge/plugin-sdk` | the compiler and its interfaces |
| `@bridge/gen-react` · `@bridge/widgets-material` | the default generator and catalog |
| `@bridge/runtime-react` | what generated applications import |

Generators and catalogs load **by name at runtime** — the first-party ones are not special-cased, and a
package you install in your project wins over the bundled one. [Writing one](docs/guide/plugins.md).
