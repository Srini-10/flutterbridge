# `@bridge/runtime-react`

The runtime compatibility kit React/Next.js applications generated by FlutterBridge target (ADR-6).

Generators emit code **against** this package; they do not inline its behaviour. That is what keeps generated
code small enough to review, and what lets a kit upgrade fix rendering bugs across every converted
application without regenerating any of them.

**Status: M3-A.** The state, theme and navigation engines are complete, and seven widgets map Flutter's
layout vocabulary onto flexbox. Flutter's constraint model underneath them, the rest of the catalog's
widgets, the `next/navigation` shim and the animation engine are not — see
[What is not here](#what-is-not-here).

## The contract, in one sentence

**Structure is data; behaviour is closures.** The kit takes descriptors — a store's shape, a token set, a
route table — and the generator lowers action bodies and derivations into real TypeScript. The kit never
sees UIR ([ADR-19](../../../docs/adr/0019-runtime-kit-consumes-descriptors-not-uir.md)).

Everything the generator will emit is hand-writable, which is why this package was built and validated
*before* the generator exists. Every test in `tests/` is hand-written.

## Quick start

```ts
import { defineStore, StoreProvider, useSignal, useStore } from '@bridge/runtime-react';

// Module scope holds a *definition*, which owns no state. This is what makes it safe (ADR-15).
export const cartStore = defineStore('cart', ({ signal, derived, action }) => {
  const items = signal<readonly Item[]>([]);
  const total = derived(() => items.get().reduce((n, i) => n + i.price, 0), 'total');
  const add = action((item: Item) => items.update((current) => [...current, item]), 'add');
  return { items, total, add };
});

function Total() {
  const cart = useStore(cartStore);
  return <span>{useSignal(cart.total)}</span>;   // re-renders only when `total` changes
}

function App() {
  // The instance is created here, once per client root / per request.
  return <StoreProvider definition={cartStore}><Total /></StoreProvider>;
}
```

## Why `defineStore` and not `new CartStore()`

Because of [ADR-15](../../../docs/adr/0015-no-module-scope-mutable-state.md), which is the one place this
project deliberately refuses to translate Flutter faithfully:

```dart
final CartStore cartStore = CartStore();   // ordinary Flutter; a privacy defect on a web server
```

In Flutter that singleton is one user. In a Next.js server process a module is shared across **every
request**, so a server-rendered component reading it can serve one user another user's cart. A definition
holds no state and an instance is created per provider, so the shape that leaks cannot be written. Proved in
[`tests/ssr.test.ts`](tests/ssr.test.ts).

## The signal graph

[ADR-20](../../../docs/adr/0020-signal-graph-evaluation-semantics.md) specifies the semantics; this package
implements them and [`tests/graph.test.ts`](tests/graph.test.ts) is their executable form. In brief:

| | |
| --- | --- |
| **R1 Glitch-freedom** | No observer sees a value computed from a mix of pre- and post-write inputs. |
| **R2 Pull, don't push** | A write marks; a `derived` computes when read, and only if an input moved. |
| **R3 Equality cutoff** | Propagation stops where values stop changing. Equality is `Object.is`. |
| **R4 Dynamic deps** | A `derived` depends on what it *read*, re-collected every run. |
| **R5 FIFO effects** | Effects run in scheduling order, once per flush. |
| **R6 Batching** | Writes in a `batch` — and every action — flush once, synchronously. |
| **R7 Fixed point** | Flush runs until the queue empties; non-termination throws `BRG4001`. |
| **R8 No cycles** | A self-reading `derived` throws `BRG4002`. |

The graph in `src/internal/state/graph.ts` has **no dependencies and no React**. React enters only at
`src/internal/react/`, over `useSyncExternalStore`, as ADR-4 requires.

## API

### State

| | |
| --- | --- |
| `signal(initial)` | Writable reactive state — `sig.Signal`. |
| `derived(compute, label?)` | Lazy, cached computed value — `sig.Derived`. Must be synchronous and pure. |
| `effect(body, label?)` | Runs now and on change; may return a cleanup. Returns a disposer. |
| `batch(body)` | One flush for many writes. Nests. |
| `untracked(body)` | Read without subscribing. |
| `subscribe(source, listener)` | Change notifications, without an initial call. |
| `defineStore(name, setup)` | Declares an `app.Store`. Holds no state; safe at module scope. |
| `instantiateStore(def, opts?)` | Creates a live store. Call per provider, never at module scope. |

### Theme

`createTheme(descriptor, options?)` resolves tokens against a reactive `brightness` signal. Colour
composition — `alphaBlend`, `stateLayer`, `elevationOverlay`, `withOpacity`, `parseColor`, `formatColor` —
is the kit's half of [ADR-13](../../../docs/adr/0013-material-color-role-derivation.md): the compiler derives
the *palette* (N10, with the same `material_color_utilities` Flutter uses), and the kit *composes* it, because
hover/pressed/elevation are functions of component state the compiler cannot know.

There is not one literal colour in this package (INV-20), and not one Material opacity constant: those are
framework metadata and must be generated from `catalog/` ([ADR-18](../../../docs/adr/0018-framework-metadata-is-never-duplicated.md)),
so composition takes the opacity as an argument.

### Widgets

ADR-6's opening sentence, made real: `Row(mainAxisAlignment: spaceBetween)` becomes
`<Row mainAxisAlignment="spaceBetween">` imported from here, **not** a bespoke flexbox `<div>` at every call
site. The alignment tables are stated once, so a kit release fixes every converted application.

| | |
| --- | --- |
| `Text` | A string, in a `<span>`. Flutter's `Text(String data)` — not a slot, so not a `ReactNode`. |
| `Column` / `Row` | `Flex` on flexbox: `mainAxisAlignment`, `crossAxisAlignment`, `mainAxisSize`. |
| `Center` | `Align(alignment: center)` — centres and expands. |
| `Padding` | `padding: EdgeInsets`, `child`. |
| `SizedBox` | `width?`, `height?`, `child?`. `Infinity` is `double.infinity` → `100%`. |
| `ElevatedButton` | `onPressed?`, `child`. **`onPressed: null` disables it** — Flutter's rule, not a default. |
| `EdgeInsets` | `.all(n)` / `.symmetric({vertical, horizontal})` / `.only({...})`, mirroring Dart. |

Single-child widgets take `child` and multi-child widgets take `children`, because
`catalog/widgets/material.json` draws exactly that line (`"slots": ["child"]` vs `"childrenProp":
"children"`) and ADR-18 makes it the only place that may.

Flutter's defaults are Flutter's, not CSS's — `crossAxisAlignment` defaults to `center` where `align-items`
defaults to `stretch`, and `mainAxisSize` defaults to `max`, which is what gives `mainAxisAlignment` free
space to distribute. `mainAxisSize: 'max'` emits a `100%` main-axis extent; under an auto-height ancestor
that degrades to shrink-wrap where Flutter would throw, which is the constraint model's job and is tagged as
deferred rather than guessed at.

There is no colour on any of them (INV-20). `ElevatedButton` is deliberately unstyled: its container,
elevation and state layers are Material roles the theme resolves at opacities that must come from `catalog/`
(ADR-18). ADR-13's evidence table *is* this widget — a hand-written reference that guessed a plausible purple
and was wrong by 15/255 on a screen that looked finished.

### Navigation

`createRouter(descriptor)` holds the route table, stack, and current entry — all reactive.

A `Destination` is a **route or a component, exactly one**, mirroring
[Spec v2.4 §A17](../../../docs/spec/v2.4-amendments.md):

```ts
router.push({ kind: 'route', route: 'product', params: { id: '42' } }); // context.go('/product/42')
router.push({ kind: 'component', component: 'HomeScreen' });            // Navigator.push(MaterialPageRoute(...))
```

A component destination has **no path, and the runtime invents none** (§A17.6): which URL an inline push
becomes on a path-based target is a legalization decision for the layer that knows the target. `pop()` is a
method, not a destination — §A17.3 is explicit that returning along an existing edge is not a transition.

### React

| | |
| --- | --- |
| `useSignal(source)` | Subscribes to a signal; re-renders on actual change. |
| `useDerived(compute, deps?)` | Component-scoped derivation. `deps` rebuild it; the graph's own deps stay dynamic. |
| `useSignalEffect(body, deps?)` | An effect that tracks signals dynamically — no dependency array to keep in sync. |
| `useMountEffect(body)` | `initState` — runs once; cleanup on unmount. |
| `useUpdateEffect(body, deps)` | `didUpdateWidget` — **skips the first render**, which `useEffect` does not. |
| `useUnmountEffect(body)` | `dispose`. |
| `StoreProvider` / `useStore` | Store scoping (ADR-15). |
| `ThemeProvider` / `useTheme` | Theme scoping. |
| `RouterProvider` / `useRouter` | Router scoping. |

The lifecycle mapping is not invented here — it is `catalog/widgets/material.json`'s `lifecycle` block
(ADR-18), which extraction has already folded into `sig.Effect.timing`.

## Diagnostics

`RuntimeError` carries a `code` field, in the `BRG4xxx` range ADR-20 claims for the runtime (the analyzer
owns `BRG13xx`, the compiler `BRG23xx`).

| | |
| --- | --- |
| `BRG4001` | A flush did not reach a fixed point — an effect reschedules itself. |
| `BRG4002` | A `derived` reads itself. |
| `BRG4003` | A store was used after `dispose()`. |
| `BRG4004` | An unknown route, or a missing path parameter. |
| `BRG4005` | A hook was used outside its provider. |
| `BRG4006` | A token the theme does not define. |
| `BRG4007` | A colour token whose value is not a colour. |

Each throws rather than degrading. A capped flush or a defaulted colour produces an application that is
subtly wrong and passes every test, which is the failure mode this project exists to avoid.

## What is not here

Tagged in `src/index.ts` as `BRIDGE-STUB`, per the repo's stub discipline:

- **Flutter's constraint model** — intrinsic sizing, and the unbounded main axis where Flutter throws and the
  widgets here degrade to shrink-wrap. They map props to style; a component can see its props and cannot see
  whether its ancestor is bounded, so solving that is a layer rather than a patch.
- **Every widget outside the seven** — the catalog lists around fifty — and the Material *appearance* of the
  one that has any, whose roles and state-layer opacities must be generated from `catalog/` rather than
  hand-written here (ADR-18).
- **The `/layout`, `/theme`, `/nav`, `/state` sub-path entrypoints.** Everything is reachable from the
  barrel; the sub-paths are a packaging decision, not a missing capability.
- **The `next/navigation` shim.** [ADR-16](../../../docs/adr/0016-nextjs-major-version.md) pins
  `next@15.5.x` as the peer range and re-decides at the M3-T6 freeze; committing to a Next surface now would
  pre-empt the decision it sequenced. The router underneath is target-neutral and complete.
- **The animation engine.** Post-MVP, per Spec v2.0 §5.

## Development

```bash
pnpm --filter @bridge/runtime-react test       # 197 tests
pnpm --filter @bridge/runtime-react bench      # not a CI gate; see tests/graph.bench.ts
pnpm --filter @bridge/runtime-react typecheck
```

Six of the eight suites run in Node with no DOM, which is the evidence that the graph, stores, theme and
router genuinely do not depend on React. Only the two that render — `react.test.ts` and `widgets.test.ts` —
opt into jsdom, with a per-file docblock, so the package still needs no `vitest.config.ts`.
